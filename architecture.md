# PY DBMS — System Design & Architecture Blueprint

> This document is designed as the ultimate blueprint of the PY DBMS project. Whether you are a non-technical reader curious about how software works, a new open-source contributor, or an engineer preparing for a system design interview, this guide will walk you through every layer of the codebase from massive global architectures down to micro-optimizations.

---

## 1. What is PY DBMS?

### The Non-Technical Explanation
Imagine you manage a giant warehouse full of filing cabinets (a Database). Usually, to ask for a file, you have to use a very ugly, boring, and sometimes confusing computer terminal provided by the warehouse company. 
**PY DBMS is a premium, custom-built front desk for that warehouse.** It doesn’t replace the warehouse, but it gives you a beautiful screen, a secure local login system, fast exports to Excel, and color-coded alerts that make asking for your files a fast, visual joy rather than a chore.

### The Engineering Explanation
PY DBMS is a **Modular Command-Line Interface (CLI) tool** built in Python 3.10+. It acts as a specialized proxy layer between a user's local terminal and a backend database engine (currently only MySQL). Its primary goal is to provide a superior Developer Experience (DX) by layering robust local security (Argon2), strict configuration hierarchies (Global, Session, Inline), and a rich UI component engine directly over standard database wire protocols.

## 2. Global Architecture: The 5 Core Pillars

At its core, PY DBMS is divided into five distinct subsystems. Structurally, it follows an internal **Layered Architecture**.

### Pillar 1: The UX / Presentation Layer (`main/`)
The visual engine of the system. 
- **The REPL (`main/core.py`)**: The Read-Evaluate-Print-Loop is the heartbeat of the application. It waits for user input, triggers the engine, and then goes back to waiting.
- **Rich Components (`main/dependencies.py`, `main/runtime.py`)**: PY DBMS leverages the Python `rich` library. We don't just print text; we construct complex and beautiful `Table`, `Panel`, `Align`, and `Group` ASCII objects. Every visual element is explicitly defined with semantic styling combinations (e.g., `[bold cyan]`, `color(57)`, custom print function to print text with typewriter effect in different colours).
- **The Centralized Output Router (`main/runtime.py` and `engine/`)**: We don't use Python's built-in `print()`. All output passes through our custom `Print()` wrapper, which enforces strict color mapping, slow-typing typewriter effects (for success messages), and instant-rendering (for errors).
- **Dual-Stream Error Handling**: Errors generated by the core CLI (like syntax violations or missing configurations) are routed through generic `pydbms_error()` mechanisms prefixed with `pydbms error>`. Conversely, failures triggered *inside* the database connection (like missing tables or bad credentials) are routed through abstract engine handlers (e.g., `MySQLErrors.error()`) and are explicitly prefixed with `<engine> error>` to help developers isolate whether a bug is local or remote.

### Pillar 2: Security & Profile Auth (`profile/`)
Before any network socket is opened to a database, PY DBMS forces a local authentication layer.
- **Argon2 Hashing**: We use the `crypto-functions` PyPI package (a standalone library by the same developer, which wraps `argon2-cffi` internally) to hash the user's master password before storing it in the OS-appropriate `pydbms/` data directory (resolved by `pydbms_path.py` — `%APPDATA%/pydbms/` on Windows, `~/Library/Application Support/pydbms/` on macOS, `~/.local/share/pydbms/` on Linux). Argon2 has built-in salting and extreme resistance to GPU cracking, helping against brute force attacks.
- **Masking (`pwinput`)**: Passwords input to the CLI are masked with asterisks (*) in real-time to prevent shoulder-surfing and to keep plain-text secrets out of your bash or zsh history files.

### Pillar 3: The Custom SQL Parser (`main/query_parse_and_classify.py`)
This is the "brain" that intercepts what a user types.
- **The State Machine**: The parser isn't a simple string `.split()`. It runs a character-by-character state machine capable of tracking whether it is currently inside a `'`, `"`, or an escape `\` delimiter. This ensures that an inline semi-colon inside a SQL string (e.g., `SELECT "hello;world"`) doesn't accidentally trigger premature query execution.
- **Flag Extraction (`classify_rest()`)**: Using Python's `shlex` library, the parser cleanly extracts arbitrary `pydbms` flags appended after the terminal semicolon (e.g., `--row-limit 50 --export json`).
- **Classifier**: Before execution, the query is statically analyzed using `startswith()` checks to classify it as a `SELECT`, `DDL`, `CHANGE`, or `META` command. This classification dictates which execution path handles it. (Note: regex is used separately in `get_query_title()` for extracting human-readable table names for the result panel title, not for classification.)

### Pillar 4: The Database Engine Abstraction (`db/` and `engine/`)
The network communication layer.
- **Abstract Base Classes (ABCs)**: Multi-engine support for the v5.x series is already baked in. `DBConnector` and `DBErrorHandler` are abstract blueprints.
- **Polymorphism**: The `connect_db` manager uses a Factory pattern. It holds a dictionary registry (`DRIVERS = {"mysql": MySQLConnector}`). The system asks the registry for a driver, prompts for credentials natively, and establishes the socket.
- **Structured Execution Result**: The DB connector's `connect()` method returns a raw `(connection, cursor)` tuple to the main REPL loop. However, the rendering layer casts cursor results into a `QueryResult` Python `dataclass` before returning them upstream. This structured result object (`QueryResult`) is what the Export pipeline consumes, effectively isolating the export drivers from raw database cursor internals.

### Pillar 5: The Export Pipeline (`export/`)
The data serialization layer.
- **The Registry Pattern (`export/export_manager.py`)**: Similar to the DB drivers, Exporters are registered in an `EXPORTERS` dictionary. When the user types `--export json`, the manager instantiates the `JSONExporter`.
- **Abstract Inheritance**: All exporters inherit from the `Exporter` base class which enforces a strictly typed `.export(result, path, include_query)` signature.
- **The Query Embedding Toggle**: A unique feature allows exporting users to append `--include-query`. The exporters use dynamic file-formatting (e.g., placing `# SELECT...` at the top of a CSV or wrapping JSON arrays in a `{query: SQL, rows: Array}` parent object) to stamp the result set with its origin command.

## 3. The Lifecycle of a Complex Request (Micro-Scale)

Let's slow time down to the microsecond. Say a user types:
`SELECT id, name FROM users; --row-limit 5000 --export csv`

1. **Wait State**: The `get_query_mysql()` loop blocks the thread, waiting for keyboard IO. As long as it hasn't seen an unquoted `;`, it prompts `    ->` on new lines.
2. **Parser Break**: Semicolon found. The raw string is passed to `parse_query_and_flags()`.
3. **Extraction**:
   - `query` = `"SELECT id, name FROM users"`
   - `rest` = `"--row-limit 5000 --export csv"`
4. **Flag Dict Generation**: `classify_rest()` splits `--row-limit` and `--export`, validates that `5000` is deeply valid (integer > 0) via a `try/except` block, and generates a structured flag dictionary.
5. **Execution Routing**: `classify_query()` determines string starts with `SELECT`. It routes execution to `execute_select()`.
6. **Network IO**: 
   - A `time.perf_counter()` captures the start time.
   - The main thread fires `cur.execute()`. 
   - Concurrently, a `rich` context manager `with console.status("Executing...")` takes over the active terminal line to draw an animated spinner, which notifies the user that the terminal is still running and has not crashed.
7. **Cursor Data Drain**: Because `--row-limit 5000` is active, the driver skips `cur.fetchall()` and explicitly calls `cur.fetchmany(5000)`. To prevent MySQL server buffer bloating, a fast `drain_remaining_rows()` loop silently pulls the rest of the cursor chunks and discards them.
8. **Object Instantiation**: The `QueryResult` dataclass is constructed.
9. **Export Interception**: The REPL notices `--export csv` is true. `ExportManager` verifies directory permissions (throwing explicit `PermissionError` if blocked), then streams the 5,000 dataclass rows to disk using `csv.writerows()`.
10. **Rich Rendering**: The dataclass is handed to the ASCII Table renderer. It maps the terminal width, calculates overflow limits, drops the data in, stops the `perf_counter()`, and prints the final execution time.

## 4. "This Was Intentional" (Design Philosophy)

*"Why did you make these technical tradeoffs?"*: A comprehensive guide to the intentional architectural design for `pydbms`.

### A. Centralized UX via the `DBErrorHandler`
**The Problem**: In early versions, `print("Error!")` was scattered across all the different files. If we wanted to change the error string from `"Error>"` to `"mysql error>"` or update its color, we had an unmaintainable codebase.
**The Intentional Solution**: We built an Abstract Base Class (ABC) named `EngineBase` alongside `DBErrorHandler` (`db/db_errors.py`). Every single piece of text that goes to the screen *must* pass through these classmethod choke points. **We use classmethods so we don't need to instantiate the error classes just to print a string.**
**The "Why"**: The primary value-prop of this tool is exceptional Developer Experience. By funneling all output through a single choke point, we guarantee that a network failure and a local syntax error look distinct but uniformly branded.

### B. The 3-Tier Configuration Hierarchy
**The Problem**: Global configurations are rigid. If users drop into a session and just want to look at one massive string without line-wrapping, they shouldn't have to rewrite their persistent `~/.pydbms/config.json`.
**The Intentional Solution**: We engineered a cascading 3-tier state system:
   1. **Global Config** (`~/.pydbms/config.json`): Persistent across reboots. Loaded once at startup.
   2. **Session Config** (`.session-config`): A Python dictionary existing only in RAM. Resets the moment you `.exit`.
   3. **Inline Flags** (`--row-limit`, `--expand`): Appended to raw SQL. Hyper-specific to *exactly* one query evaluation loop. 
**The "Why"**: This provides maximum flexibility without mutating global state inappropriately. The parser logic handles precedence mapping, always prioritizing the most granular flag.

## 5. "It Naturally Grew Into This" (Organic Evolution)

Some of the strongest subsystems evolved directly out of initial pain points:

- **The State Machine Parser**: On day one, to find a semi-colon, we just did `if raw_query.endswith(";")`. But what if a user typed `SELECT "hello;world" FROM table`? Or added standard SQL comments at the end? The simple check shattered. The parser organically evolved into the current `O(n)` linear scan state machine that tracks delimiters (`""`, `''`) and escapes (`\`) to split accurately.
- **The Flag Argument Extractor**: We originally used Python's `argparse` standard library to pick up inline query flags. However, `argparse` assumes you are feeding it bash commands, and throws fatal `SystemExit` signals on invalid syntax — intentionally crashing our entire DB session! We had to completely abandon it and organically write a custom `shlex` validator inside `classify_rest()`.

## 6. System Bottlenecks & Hard Limits

A senior engineer knows perfectly where their system cracks. Here are the explicit failure points of PY DBMS v4.1.x:

### 1. The `fetchall()` Memory Bomb
**The Architecture**: When running a raw query without limits (e.g. `SELECT * FROM logs`), PY DBMS issues a `cur.fetchall()` command at the driver level. This attempts to pull the *entire* database payload out of the network buffer and map it as Python Lists into local RAM.
**The Limit**: If the dataset contains say 25 million rows, the Python process will rapidly consume gigabytes of memory, resulting in an OS-level Out-Of-Memory (OOM) kill.
**The Future Fix**: Iterative chunking. We must transition the core to yield rows dynamically (`fetchmany(1000)` in a while loop), streaming them directly to the terminal buffers or CSV filesystem layer, completely bypassing the `QueryResult` memory wrapper for massive payloads.

### 2. Python's Global Interpreter Lock (GIL)
**The Architecture**: Standard CPython employs the GIL, drastically limiting true multithreading on CPU-bound workloads. 
**The Limit**: While PY DBMS cleverly spins up a background thread using `rich.console.status` for the visual "Loading..." spinner so the UI doesn't freeze, the actual generation of massive JSON exports on the main thread will lock the core process entirely. We cannot currently run query fetching and file writing genuinely parallelized due to the GIL.

### 3. CI/CD Stalling (The Interactive Prompt Trap)
**The Architecture**: We use the excellent `pwinput` library in `profile/profile_auth.py` and `db/mysql.py` to mask passwords natively with `*`. 
**The Limit**: When executing automated test suites (like `pytest`) in non-interactive CI/CD pipelines (like GitHub Actions), the runtime reaches `pwinput.pwinput()`, the terminal locks entirely waiting for standard input from an invisible user, and the entire test pipeline stalls permanently. The fix requires heavy use of `mock.patch` objects in the test suite to bypass interactive prompts.

### 4. Terminal Dynamism 
**The Architecture**: We calculate column overflow truncations mathematically based on `os.get_terminal_size()`. 
**The Limit**: If a user actively drags to resize their graphical terminal window *during* the sub-second calculation of a massive rendered table, the mathematical constraints map against stale terminal dimensions. The result is a visually corrupted, jagged ASCII table output for that single command.

## 7. The Road Forward (v5.x Pipeline)

With the completion of v4.1.0, the core execution environment is stable. The v5.x initiative revolves entirely around **Multi-Database Support (Polymorphism)**.

Because we have meticulously established the `DBConnector` abstract base classes, the generic `DatabaseError` exception mapping, and the `DBErrorHandler` output mechanisms in v4.1.0, to add **PostgreSQL** or **SQLite** support we no longer need to touch the parser, the UI loop, or the exporter.
We simply:
1. Create `db/postgres.py` implementing `DBConnector`.
2. Map `psycopg2` exceptions to our internal `DatabaseError`.
3. Add it to the `connect_db.DRIVERS` registry.

This perfectly clean architectural boundary proves that PY DBMS has matured from a simple script into a highly scalable Python framework.
